# Пользовательские сценарии

Этот документ описывает текущие рабочие сценарии приложения с точки зрения пользователя и соответствующих backend-вызовов.

## Генерация точек

Пользователь вводит:

- количество точек;
- широту центра;
- долготу центра;
- радиус в километрах.

Frontend валидирует значения и вызывает `POST /points/generate`. Backend очищает текущие точки и маршруты, генерирует новые точки внутри круга и сохраняет их в SQLite.

После успешного ответа frontend:

- кладёт точки в `store.points`;
- сбрасывает `baseRoute` и `optimizedRoute`;
- отрисовывает маркеры на карте;
- сбрасывает метрики.

## Построение базового маршрута

Пользователь нажимает «Построить маршрут». Frontend берёт ID точек из текущего store и вызывает `POST /routes/base`.

Backend:

- достаёт точки в переданном порядке;
- вызывает текущий routing provider;
- сохраняет маршрут, координаты, геометрию, расстояние, время и metadata provider;
- возвращает маршрут frontend.

По текущим env-шаблонам основной provider - `osrm`, поэтому базовый маршрут строится по дорожной geometry. Если OSRM недоступен, backend возвращает `haversine` fallback с прямыми линиями.

## Оптимизация маршрута

Пользователь нажимает «Оптимизировать». Frontend вызывает `POST /routes/optimize` с теми же ID точек.

Backend:

- загружает точки;
- сохраняет стартовую точку первой;
- строит порядок обхода через Nearest Neighbor;
- передаёт оптимизированный порядок routing provider;
- сохраняет и возвращает новый маршрут.

Сейчас 2-opt не реализован: оптимизация ограничена алгоритмом ближайшего соседа.

## Сравнение метрик

Когда в store есть и базовый, и оптимизированный маршрут, frontend считает разницу через `frontend/js/features/compare-metrics.js` и отображает:

- экономию в километрах;
- экономию во времени;
- процент улучшения;
- отдельное сообщение, если оптимизированный маршрут оказался длиннее.

## Сброс точек

Пользователь нажимает «Сбросить точки». Frontend вызывает `DELETE /points`.

Backend удаляет все точки и все маршруты. Frontend очищает store, карту, карточки метрик и список порядка посещения.

## Поведение при ошибках

Frontend показывает уведомления через `frontend/js/ui/notifications.js` и меняет `status` в store. Кнопки блокируются во время loading-состояний и при недостаточном количестве точек.

Fallback-предупреждение в UI предусмотрено для маршрутов с `is_fallback: true`. Factory оборачивает OSRM provider в `RoutingRouter`, поэтому fallback на haversine является частью основного provider-пути.
