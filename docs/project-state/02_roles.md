# Роли в проекте

В проекте есть не только пользователи, но и технические роли. Каждая часть системы отвечает за отдельный кусок смысла.

## Пользователь / диспетчер

Пользователь задаёт условия маршрута: центр, радиус и количество точек. Затем он строит базовый маршрут, запускает оптимизацию и сравнивает результат.

Его главный интерес - не API и не алгоритмы, а ответ на вопрос: какой порядок посещения точек даёт более короткий маршрут.

## Frontend

Frontend - это карта и панель управления. Он отвечает за:

- ввод параметров генерации;
- валидацию формы;
- вызовы backend API;
- отображение точек, линий маршрута и порядка посещения;
- визуальное различение базового и оптимизированного маршрута;
- показ ошибок, loading-состояний и метрик.

Ключевые файлы находятся в `frontend/js/`: `app.js`, `state/store.js`, `api/client.js`, `features/`, `map/`, `ui/`.

## Backend API

Backend - координатор сценариев. Он принимает HTTP-запросы от frontend, вызывает сервисный слой, работает через repositories и unit of work, сохраняет результат в SQLite.

Основные роутеры:

- `backend/api/points.py` - генерация, получение и очистка точек;
- `backend/api/routes.py` - построение, оптимизация и чтение маршрутов;
- `backend/api/system.py` - health и config.

## Сервисы маршрутизации

Сервисный слой - «мозг» маршрута. Он решает, какие точки взять, в каком порядке их передать routing provider и какие данные сохранить.

Главные роли:

- `backend/services/point.py` - генерация и очистка точек;
- `backend/services/route.py` - построение и оптимизация маршрутов;
- `backend/services/routing_service.py` - алгоритм ближайшего соседа;
- `backend/services/routing_providers/` - источники расчёта расстояния и геометрии.

## SQLite

SQLite - рабочая память проекта. В ней живут две сущности:

- `points` - координаты точек;
- `routes` - порядок точек, метрики, геометрия, provider и дополнительные признаки маршрута.

Для демо и локальной разработки этого достаточно. Для многопользовательского production-сценария такой подход потребовал бы расширения модели данных.

## Тесты

Тесты выполняют роль защитного слоя. Они проверяют алгоритмы, provider-логику, API-обёртки и repositories.

Текущий автоматический прогон проходит: `50 passed` при `python3 -m pytest tests/ -q`. OSRM provider реализует `build_route`, а fallback-поведение проверяется отдельными unit-тестами.
