# Архитектура

Приложение — это два контейнера и одна база SQLite. Снаружи всё выглядит как один сайт: запросы идут в nginx, а он либо отдаёт статику, либо проксирует их в бэкенд.

```text
браузер
  -> nginx (фронтенд)
    -> статические файлы
    -> прокси /api
      -> FastAPI
        -> сервисы
        -> репозитории
        -> SQLite
        -> провайдер маршрутов
        -> провайдер геокодинга
```

## Фронтенд

Фронтенд лежит в `frontend/` и отдаётся nginx как набор статических файлов. Сборщика и фреймворка тут нет — это обычные HTML, CSS и ES-модули:

- `frontend/index.html` — разметка страницы;
- `frontend/css/styles.css` — стили;
- `frontend/js/app.js` — точка входа;
- `frontend/js/api/client.js` — HTTP-клиент к бэкенду;
- `frontend/js/features/` — действия пользователя: генерация, геокодинг, построение, оптимизация, очистка;
- `frontend/js/map/` — карта MapLibre, маркеры и линии маршрутов;
- `frontend/js/state/store.js` — простое общее состояние.

В API фронтенд ходит по адресу из `FRONTEND_API_BASE_URL`. В Docker это `/api`, а сам проксированием занимается nginx.

Адреса ищутся только по явному действию пользователя: отдельная кнопка для центра генерации и отдельная — для ручной точки. Автодополнения нет, импортированные файлы пачкой не геокодируются. Если провайдер вернул несколько кандидатов, выбор остаётся за пользователем.

### Экономия в рублях

Этот блок целиком живёт на фронтенде, отдельной ручки в бэкенде под него нет. Когда построены и базовый, и оптимизированный маршруты, фронтенд просто сравнивает уже полученные метрики и показывает:

- сколько километров сэкономлено;
- сколько минут сэкономлено;
- сколько это примерно в рублях.

Рубли считаются по коэффициентам из интерфейса — `₽/км` и `₽/мин`, по умолчанию `35` и `8`. Если коэффициент пустой, отрицательный или не число, он считается нулём.

## Бэкенд

Бэкенд лежит в `backend/` и запускается как приложение FastAPI. Он разбит на слои:

- `backend/api/` — HTTP-эндпоинты;
- `backend/schemas/` — Pydantic-схемы запросов и ответов;
- `backend/services/` — логика генерации точек, построения и оптимизации маршрутов;
- `backend/repositories/` — работа с таблицами через сессию SQLAlchemy;
- `backend/db/` — модели, фабрика сессий и Unit of Work;
- `backend/domain/` — простые доменные модели.

Файл `backend/main.py` собирает всё вместе: подключает CORS, инициализирует базу и регистрирует роутеры.

## Данные

База — SQLite, по умолчанию файл `data/database.db`. Основные таблицы:

- `points` — точки: `id`, `lat`, `lon` и необязательные поля адреса;
- `routes` — маршруты: порядок точек, координаты, метрики, геометрия и информация о провайдере;
- `route_shares` — снимки маршрутов для публичных ссылок;
- `geocode_cache` — кэш ответов геокодера, чтобы не дёргать его повторно.

Генерация нового набора точек стирает старые точки и маршруты — это сделано намеренно, приложение работает с одним текущим набором. Ручное добавление точки набор расширяет, но сохранённые маршруты тоже сбрасывает: после изменения состава точек прежние маршруты уже неактуальны.

Адреса хранятся в `points.address`, `points.geocoding_provider` и `points.geocoding_place_id`. У сгенерированных, импортированных и поставленных кликом точек адреса может не быть — тогда интерфейс показывает координаты.

## Маршруты

Базовый маршрут строится ровно в том порядке, в котором точки пришли от клиента.

Оптимизированный — одним из двух способов:

- **ближайший сосед**: берём первую точку, каждый раз идём в ближайшую из ещё не посещённых, пока точки не закончатся;
- **2-opt**: стартуем с маршрута «ближайшего соседа» и разворачиваем участки пути, пока это укорачивает общую длину.

Оба способа — быстрые эвристики и не гарантируют самый короткий маршрут в мире, но заметно лучше исходного порядка.

Сравнение маршрутов для интерфейса считается на клиенте: фронтенд берёт метрики обоих маршрутов из состояния и показывает разницу по расстоянию, времени, проценту улучшения и примерной экономии в рублях.

## Провайдеры маршрутов

Основной провайдер задаётся через `ROUTING_PROVIDER`. В обычном режиме это OSRM:

- бэкенд отправляет координаты в OSRM;
- OSRM возвращает расстояние, время и геометрию по дорогам;
- фронтенд рисует маршрут по дорогам.

Если основной провайдер падает, `RoutingRouter` переключается на резервный расчёт по прямой (haversine):

- расстояние — по прямой между координатами;
- время — из фиксированной скорости 40 км/ч;
- геометрия — прямые линии между точками;
- в ответе выставляется `is_fallback=true`.

## Провайдер геокодинга

Геокодинг идёт через бэкенд (`POST /geocode` и `POST /geocode/reverse`), чтобы фронтенд не ходил во внешний сервис напрямую. По умолчанию используется Nominatim-совместимый сервис:

- `GEOCODING_PROVIDER=nominatim`;
- `GEOCODING_BASE_URL=https://nominatim.openstreetmap.org`;
- `GEOCODING_TIMEOUT_SECONDS=10`;
- `GEOCODING_USER_AGENT=route-optimization-demo/1.0`;
- `GEOCODING_ACCEPT_LANGUAGE=ru,en`.

Бэкенд нормализует ответы, кэширует их в SQLite и ограничивает исходящие запросы одним в секунду на всё приложение. Прямой геокодинг используется для поиска центра и ручной точки, обратный — чтобы подписать координаты адресом. Массового геокодинга при генерации и импорте нет специально, чтобы не упираться в лимиты провайдера.

## Docker Compose

Локальный `docker-compose.yml` поднимает два сервиса:

- `backend` — FastAPI, порт из `BACKEND_PORT`, том `./data:/app/data`;
- `frontend` — nginx, порт из `FRONTEND_PORT`, проксирует `/api` в бэкенд.

Фронтенд стартует только после того, как health-проверка бэкенда прошла успешно.
