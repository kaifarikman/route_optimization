# API

Бэкенд отдаёт JSON. Локально он доступен напрямую на `http://localhost:8000` и через прокси фронтенда на `http://localhost:8080/api`. Живая документация со всеми схемами — Swagger UI на `http://localhost:8000/docs`.

## Общие правила

Тело запроса и ответа — всегда `application/json`. Ошибки приходят в стандартном виде FastAPI:

```json
{
  "detail": "Текст ошибки"
}
```

Чаще всего встречаются такие статусы:

- `200 OK` — запрос выполнен;
- `400 Bad Request` — некорректный запрос или не нашлись точки для маршрута;
- `404 Not Found` — нет маршрута или ссылки с таким идентификатором;
- `422 Unprocessable Entity` — тело не прошло валидацию схемы;
- `429 Too Many Requests` — превышен лимит геокодинга;
- `502` / `504` — провайдер геокодинга недоступен или не ответил вовремя;
- `500 Internal Server Error` — ошибка чтения или записи на стороне бэкенда.

## Модели

### Point

```json
{
  "id": 1,
  "lat": 55.751244,
  "lon": 37.618423,
  "address": "Москва, Красная площадь",
  "geocoding_provider": "nominatim",
  "geocoding_place_id": "123"
}
```

Поля `address`, `geocoding_provider` и `geocoding_place_id` необязательны. У сгенерированных, импортированных и поставленных кликом точек их обычно нет.

### Route

```json
{
  "id": 1,
  "points": [1, 3, 2],
  "distance_km": 12.4,
  "duration_minutes": 23.8,
  "coordinates": [
    [55.751244, 37.618423],
    [55.761244, 37.628423],
    [55.741244, 37.608423]
  ],
  "geometry": [
    [55.751244, 37.618423],
    [55.755000, 37.620000],
    [55.761244, 37.628423]
  ],
  "provider": "osrm",
  "is_fallback": false,
  "geometry_type": "full",
  "transport_type": "driving"
}
```

Что значат поля маршрута:

- `points` — ID точек в порядке посещения;
- `coordinates` — координаты этих точек в том же порядке;
- `geometry` — линия, которую фронтенд рисует на карте;
- `provider` — `osrm` или `haversine`;
- `is_fallback` — `true`, если основной провайдер упал и использован резервный расчёт;
- `geometry_type` — `full` для дорожной геометрии OSRM, `straight` для прямых линий;
- `transport_type` — сейчас всегда `driving`.

## System

### `GET /health`

Проверяет, что бэкенд жив.

```json
{
  "status": "ok",
  "message": "Сервер запущен"
}
```

### `GET /config`

Возвращает настройки, которые нужны фронтенду и проверкам.

```json
{
  "routing_api": "osrm",
  "version": "1.0.0",
  "cors_enabled": true,
  "database": "sqlite"
}
```

## Geocoding

### `POST /geocode`

Ищет координаты по адресу — бэкенд сам ходит в Nominatim-совместимый сервис.

Запрос:

```json
{
  "query": "Москва, Красная площадь",
  "limit": 5
}
```

Ограничения: `query` — от 3 до 300 символов, `limit` — от 1 до 5.

Ответ:

```json
{
  "results": [
    {
      "lat": 55.75393,
      "lon": 37.6208,
      "display_name": "Красная площадь, Тверской район, Москва, Россия",
      "provider": "nominatim",
      "place_id": "123",
      "category": "place",
      "type": "square",
      "importance": 0.9
    }
  ]
}
```

Если кандидатов несколько, фронтенд показывает список и не выбирает адрес за пользователя. Если ничего не нашлось, `results` будет пустым.

Ошибки провайдера:

- `429` — локальный лимит «один запрос в секунду» или лимит на стороне сервиса;
- `502` — провайдер вернул мусор или недоступен;
- `504` — провайдер не ответил вовремя.

### `POST /geocode/reverse`

Обратная операция: по координатам возвращает адрес. Используется, чтобы подписать точку человекочитаемым названием.

Запрос:

```json
{
  "lat": 55.75393,
  "lon": 37.6208
}
```

Ограничения: `lat` — от −90 до 90, `lon` — от −180 до 180.

Ответ — один результат той же формы, что и в `/geocode`, либо `null`, если адрес не нашёлся:

```json
{
  "result": {
    "lat": 55.75393,
    "lon": 37.6208,
    "display_name": "Красная площадь, Тверской район, Москва, Россия",
    "provider": "nominatim",
    "place_id": "123",
    "category": "place",
    "type": "square",
    "importance": 0.9
  }
}
```

Лимит и кэш — те же, что у прямого геокодинга.

## Points

### `POST /points/generate`

Генерирует новый набор точек вокруг центра.

Важно: генерация заменяет текущий набор — перед созданием новых точек бэкенд удаляет старые точки и сохранённые маршруты.

Запрос:

```json
{
  "center_lat": 55.751244,
  "center_lon": 37.618423,
  "radius_km": 10,
  "count": 5
}
```

Ограничения: `center_lat` ∈ [−90, 90], `center_lon` ∈ [−180, 180], `radius_km` ∈ (0, 50], `count` ∈ [2, 50].

Ответ:

```json
{
  "points": [
    { "id": 1, "lat": 55.754100, "lon": 37.612300 }
  ]
}
```

### `POST /points`

Добавляет одну точку в текущий набор.

Важно: ручное добавление точки не стирает остальные, но сбрасывает сохранённые маршруты — после изменения набора их нужно строить заново.

Запрос (поля адреса необязательны, запрос только с координатами тоже валиден):

```json
{
  "lat": 55.751244,
  "lon": 37.618423,
  "address": "Москва, Красная площадь",
  "geocoding_provider": "nominatim",
  "geocoding_place_id": "123"
}
```

Ограничения: `lat` ∈ [−90, 90], `lon` ∈ [−180, 180].

Ответ:

```json
{
  "point": {
    "id": 2,
    "lat": 55.751244,
    "lon": 37.618423,
    "address": "Москва, Красная площадь",
    "geocoding_provider": "nominatim",
    "geocoding_place_id": "123"
  }
}
```

### `POST /points/import`

Загружает сразу набор точек (например, из JSON- или CSV-файла на фронтенде). Как и генерация, заменяет текущий набор и сбрасывает маршруты. Если точек больше 50, берутся первые 50.

Запрос:

```json
{
  "points": [
    { "lat": 55.751244, "lon": 37.618423 },
    { "lat": 55.761244, "lon": 37.628423, "address": "Точка 2" }
  ]
}
```

Ответ — список добавленных точек в формате `Point`:

```json
{
  "points": [
    { "id": 1, "lat": 55.751244, "lon": 37.618423 },
    { "id": 2, "lat": 55.761244, "lon": 37.628423, "address": "Точка 2" }
  ]
}
```

### `GET /points`

Возвращает все сохранённые точки:

```json
{
  "points": [
    { "id": 1, "lat": 55.754100, "lon": 37.612300 }
  ]
}
```

### `DELETE /points`

Удаляет все точки и все маршруты:

```json
{
  "status": "cleared",
  "message": "Удалено точек: 5"
}
```

## Routes

### `POST /routes/base`

Строит маршрут в том порядке, в котором переданы `point_ids`.

Запрос:

```json
{
  "point_ids": [1, 2, 3]
}
```

Ограничения: `point_ids` — от 2 до 50 идентификаторов.

Ответ:

```json
{
  "route": {
    "id": 1,
    "points": [1, 2, 3],
    "distance_km": 12.5,
    "duration_minutes": 18.75,
    "coordinates": [
      [55.751244, 37.618423],
      [55.761244, 37.628423],
      [55.741244, 37.608423]
    ],
    "geometry": [
      [55.751244, 37.618423],
      [55.761244, 37.628423],
      [55.741244, 37.608423]
    ],
    "provider": "osrm",
    "is_fallback": false,
    "geometry_type": "full",
    "transport_type": "driving"
  }
}
```

### `POST /routes/optimize`

Перестраивает маршрут по тем же точкам в более выгодном порядке.

Алгоритм выбирается полем `algorithm`:

- `nearest_neighbor` (по умолчанию) — начинает с первой переданной точки и каждый раз идёт в ближайшую ещё не посещённую;
- `two_opt` — стартует с маршрута «ближайшего соседа» и разворачивает участки пути, пока это укорачивает общую длину.

Оба — быстрые эвристики, а не поиск глобального оптимума.

Запрос:

```json
{
  "point_ids": [1, 2, 3],
  "algorithm": "two_opt"
}
```

Ограничения: `point_ids` — от 2 до 50 идентификаторов; `algorithm` — `nearest_neighbor` или `two_opt`.

Ответ — такой же `route`, как у `/routes/base`, но с переставленным порядком точек:

```json
{
  "route": {
    "id": 2,
    "points": [1, 3, 2],
    "distance_km": 10.2,
    "duration_minutes": 15.3,
    "coordinates": [
      [55.751244, 37.618423],
      [55.741244, 37.608423],
      [55.761244, 37.628423]
    ],
    "geometry": [
      [55.751244, 37.618423],
      [55.741244, 37.608423],
      [55.761244, 37.628423]
    ],
    "provider": "osrm",
    "is_fallback": false,
    "geometry_type": "full",
    "transport_type": "driving"
  }
}
```

### `GET /routes/{route_id}`

Возвращает сохранённый маршрут по ID в том же формате `{ "route": { ... } }`. Если маршрута нет — `404`.

### `GET /routes`

Возвращает все сохранённые маршруты и их количество:

```json
{
  "routes": [ { "id": 1, "points": [1, 2, 3] } ],
  "total": 1
}
```

(в каждом элементе — полная модель `Route`, здесь поля сокращены для краткости.)

### `POST /routes/share`

Создаёт публичную ссылку на пару «базовый + оптимизированный» маршрут. Бэкенд складывает снимок (точки и оба маршрута) и выдаёт токен.

Запрос:

```json
{
  "base_route_id": 1,
  "optimized_route_id": 2
}
```

Ответ:

```json
{
  "share_url": "https://route-optimization.ru/?share=AbCdEf...",
  "token": "AbCdEf..."
}
```

### `GET /routes/share/{token}`

Возвращает сохранённый снимок по токену. Если ссылки нет — `404`.

```json
{
  "share": {
    "version": 1,
    "points": [
      { "id": 1, "lat": 55.751244, "lon": 37.618423, "address": null }
    ],
    "base_route": { "points": [1, 2, 3], "distance_km": 12.5 },
    "optimized_route": { "points": [1, 3, 2], "distance_km": 10.2 }
  }
}
```

Внутри снимка точки перенумерованы по порядку (1, 2, 3…), а `points` в маршрутах ссылаются на эти номера, а не на исходные ID из базы.

## Маршрутизация и резервный расчёт

По умолчанию бэкенд использует OSRM:

```env
ROUTING_PROVIDER=osrm
OSRM_BASE_URL=https://router.project-osrm.org
```

Если OSRM недоступен, маршрут считается по прямой (haversine). Такой ответ приходит с `is_fallback=true`, `provider="haversine"` и `geometry_type="straight"`.

## Геокодинг и лимиты

По умолчанию используется Nominatim-совместимый сервис:

```env
GEOCODING_PROVIDER=nominatim
GEOCODING_BASE_URL=https://nominatim.openstreetmap.org
GEOCODING_TIMEOUT_SECONDS=10
GEOCODING_USER_AGENT=route-optimization-demo/1.0
GEOCODING_ACCEPT_LANGUAGE=ru,en
```

Ответы геокодера кэшируются в SQLite, а исходящие запросы ограничены одним в секунду на всё приложение — этого требует политика публичного Nominatim. Фронтенд вызывает `/geocode` только по кнопке поиска адреса; автодополнения и пакетного геокодинга при импорте нет.
